# GMEPay+ Architecture Audit — CTO Second Pass (post-Operations backend)

**Date:** 2026-07-01 · **Lens:** what the ops build newly EXPOSED architecturally.
**Round-1 (audit_cto_2026-07-01.md, group 4.20) still unbuilt — not repeated here except the 2-3 that now gate go-live.**

The Operations wave is well-engineered *within each service* — idempotent kill-switch commands, hash-chain audited operator actions, config-gated sweepers, degrade-never-500 control tower. But it was bolted onto an event/scheduling/eventing spine that was never finished, and in doing so it **turned three latent round-1 gaps into active correctness/safety bugs** and added one new structural coupling. The through-line is the same round-1 pattern — *provisioned, not plumbed* — now on the ops surface.

---

## (a) ARCHITECTURAL GAPS THE OPS BUILD NEWLY EXPOSED

### 1. Two parallel event lineages — operator terminal states are invisible to money consumers (root cause of Finance G1/G2)
Verified in `transaction-mgmt/.../statemachine/TransactionStateMachine.java`: `transition()` **always** emits the FSM `TransactionStatusChangedEvent`, and emits the revenue-bearing `PaymentApprovedEvent` + `TransactionCommittedEvent` **only on the `APPROVED` branch**. Operator force-resolve (`TransactionService.resolveByOperator`) maps `COMPLETED→APPROVED` (OK — rides the APPROVED branch) but **`REVERSED→REVERSED` emits ONLY the FSM event**. There is no `payment.reversed` / prefund-release domain event at all. So an operator reversing a stuck UNCERTAIN txn releases held prefunding and moves money in the FSM, but revenue-ledger and prefunding **never react** — G1/G2 are a direct consequence of the FSM-vs-domain event split, not a Finance coding slip.
**This is mine to own, not Finance's.** The fix is architectural: a single **outbox-backed state→domain-event mapping** so that *every* terminal transition (whatever drives it — scheme, recon, operator) deterministically emits the domain event its consumers need. Today domain events are hand-emitted inline in the state machine for one status; that does not compose. Options: (i) a status→domain-event table applied uniformly in `transition()`, or (ii) consumers subscribe to the FSM `TransactionStatusChanged*` lineage and derive money effects themselves — pick one lineage, not both. **Terminal-state event completeness (`REVERSED`, `FAILED`-after-UNCERTAIN, `REFUNDED`) is the real gap.**

### 2. Hot-path synchronous coupling with a fail-OPEN kill switch — a safety inversion
`payment-executor` `OperationalGate.checkNewAuthorization()` calls `OperationalStatusClient.currentStatus()` on **every new authorization**, backed by `RestOperationalStatusClient` → synchronous `GET /v1/ops/operational-status` on config-registry, 3s cache, **`fail-open=true` default**. Two structural problems:
- **New synchronous dependency on the pay hot path.** Every authorization now has a soft dependency on config-registry availability/latency. Round-1 flagged "no resilience4j on inter-service calls" (#6) — this gate is exactly such a call, with **no circuit breaker / timeout budget** shown, added to the most latency-sensitive path.
- **Fail-open defeats the kill switch's purpose.** A kill switch exists to *stop* money movement in an emergency (compromised/suspended partner, scheme incident). Defaulting to fail-open means **during a config-registry outage a suspended or compromised partner transacts freely** — precisely the scenario the switch was built for. The last-known-good cache mitigates a blip, but a cold executor (post-deploy, scale-out) with no cached value + a registry outage silently allows everything.
**Recommendation:** invert the coupling — **push the flag, don't pull it per-authorization.** Publish operational-status changes as an event (or Redis-backed pushed cache) that executors hold locally; the pay path reads local state (truly non-blocking, no per-txn RPC). For the residual unreachable case, **fail-CLOSED on suspensions/global-pause is the safety-correct default** for a kill switch; fail-open is defensible only for the *availability* flags, not the *security* ones. At minimum, split the policy: never fail-open past a partner/scheme suspension.

### 3. New `@Scheduled` sweepers × no distributed lock — now a correctness bug
Confirmed **zero ShedLock** anywhere in `code/services` (no `net.javacrumbs`, `@SchedulerLock`). The ops wave added `StuckTransactionAlertSweeper`, `WebhookBacklogMonitor`, `ReconBreakAlerter` (+ existing outbox publishers, expiry sweepers, ZeroPay/settlement batch generators — all cron `@Scheduled`). On ≥2 replicas **every one double-fires**: duplicate `ops.alert`s, duplicate stuck-txn scans, and — worse than tidiness — the settlement/ZeroPay **batch file generators and outbox publishers can double-produce**. Round-1 treated single-replica as the HA gap (#2); the moment ops adds alerting AND the moment we scale to the HA target, unguarded schedulers become duplicate-work/duplicate-side-effect bugs. ShedLock (or leader election) is now table-stakes, not cleanup.

### 4. Alert pipeline is a half-built event — emit-only, no consumer, no schema governance
Confirmed: `OpsAlertPayload` (lib-api-contracts) + emitters in three services publishing to `gmepay.ops.alert`, and **zero `@KafkaListener` / consumer** for it anywhere. Alerts go to the outbox→Kafka (or just get logged by `LoggingEventPublisher` when no broker) and **nobody routes them to a pager/Slack/dashboard**. This is the round-1 "provisioned-not-plumbed" pattern reincarnated: the sweepers report "done" but the operational value (someone gets woken up) does not exist. Compounding it, the new event types (`OpsAlertPayload`, `OperationalStatusView`) ride the **same unversioned plain-JSON path** as round-1 #4 — no schema-registry binding, no version field — so the ops contract is already accreting ungoverned event shapes.

### 5. Ops-BFF control tower fan-out — a new latency/failure aggregation point
`ControlTowerController` fans out **synchronously to 7 upstreams** (transactions, webhooks, prefunding [+ per-partner balance N+1 loop], system-health, settlements, config-registry, ops-control) to compose one page. It degrades gracefully (per-section try/catch → `degradedSections`), which is good, but structurally it's a **serial fan-out with an unbounded per-partner inner loop** and no timeout budget — tail latency is the sum of the slowest reachable upstream, and it couples the operator's situational view to the health of 7 services at once. Acceptable for an internal read-only page, but it needs **parallel fan-out + per-call timeouts** before it's the tool on-call reaches for during an incident (i.e. exactly when those upstreams are unhealthy). Minor vs 1-4, but same resilience-gap family as #2.

**Also noted (lower):** operator-action idempotency in `OpsControlService` is well done (state-change-gated audit, no replay spam) — good pattern; the analogous idempotency on `resolveByOperator` is correct too. The remaining risk there is purely the missing domain event (#1), not the command semantics.

## (b) THE 2-3 UNBUILT 4.20 ITEMS THAT ARE GENUINE GO-LIVE GATES

1. **Event-contract governance + versioning (round-1 #4).** Now non-negotiable: the ops wave added new event types (`ops.alert`) onto an unversioned JSON bus feeding revenue-ledger, prefunding, webhooks and reporting. With the #1 lineage split unresolved, a payment can move money with the wrong/absent event. Bind events to the schema registry (already in compose/Helm) + add a version field, and **unify the state→domain-event mapping**. Gate: no go-live while a terminal money transition can emit an event no consumer receives.

2. **Resilience + distributed-scheduling hardening (round-1 #6 + #2/HA, now sharpened by ops).** Two concrete, buildable blockers: (a) **ShedLock/leader election on all `@Scheduled` jobs** before running >1 replica (else duplicate batches/alerts/outbox sends); (b) **circuit-breaker + timeout budget on the new synchronous operational-status gate** on the pay hot path, and **fix the fail-open→fail-closed default for the security flags** (§2). These are the difference between "ops build works on one box" and "ops build is safe under HA."

3. **Observability: correlation IDs + real alerting sink (round-1 #5).** The ops wave *emits* alerts but there is no tracing, no correlation ID across gateway→executor→txn-mgmt→ledger, and **no consumer to turn `ops.alert` into a page**. An operations control plane with no downstream alerting and no cross-service trace is a dashboard, not operability. Gate: at least a consumer that delivers `CRITICAL` alerts to a real channel + correlation-ID propagation on the payment path.

**Not repeating** round-1 DR, secrets/PII, scheme-adapter SDK, analytics/CDC, data-residency, NFR/load — still open, still important, but the three above are what the *ops build itself* now makes load-bearing for go-live.
