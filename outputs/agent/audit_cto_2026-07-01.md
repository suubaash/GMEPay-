# GMEPay+ Engineering & Architecture Audit — CTO View
**Date:** 2026-07-01 · **Lens:** Scale-out · Reliability · Data · Tech strategy
**Scope:** genuine gaps only; excludes what is already built (microservice-per-domain, outbox→Kafka, ADR-016 routing, idempotency on payment path, HMAC/OIDC/RBAC/hash-chained audit, cloud-agnostic Helm, scheme adapters behind a common interface, sandbox sims, gated REST clients).

The platform is architecturally sound in its *shape* — clean domain boundaries, event-driven spine, portability discipline. The gaps below are almost all about **operating it at volume across many partners/schemes/countries**, not about the domain model. A recurring pattern: infrastructure is *declared* (Helm env vars, docker-compose services, ADRs) but **not wired into the running code**. This "provisioned-but-not-plumbed" illusion is the single biggest risk because it reads as "done" on a status board while failing under real load.

---

## TOP ENGINEERING / ARCHITECTURE GAPS

### 1. No Scheme-Adapter SDK or certification harness — the core scaling axis is manual
Each scheme is a **bespoke microservice**, not a plug-in. ZeroPay exposes a rich `SchemeAdapter` interface (`services/scheme-adapter-zeropay/.../adapter/SchemeAdapter.java`); Nepal implements a *different, simpler* surface (`NepalSchemeAdapter`, decode/submit/status). The orchestrator's real contract is `SchemeClient` (`payment-executor/.../domain/client/SchemeClient.java`) dispatched by a **hard-coded router** (`SchemeClientRouter.java`) keyed on scheme_id string — adding a scheme means editing Java + recompiling. There is **no shared `SchemeAdapterPort` base, no registry/factory, no conformance test kit, no "how to add a scheme" doc/ADR.** Effort for scheme #3: ~2–3 wks (REST-style) to 4–6 wks (batch/SFTP-style).
*Why it matters:* This is the growth engine. Onboarding country #5 or scheme #8 currently scales in engineer-weeks and core-service recompiles, not config. Without an SDK + certification harness (golden fixtures, contract suite every adapter must pass), each new partner is a bespoke integration risk and a regression surface on the payment path.

### 2. No horizontal availability — `defaultReplicas: 1`, single DB per service
Helm ships `defaultReplicas: 1` (`deploy/helm/gmepay/values.yaml`) with **no HPA, no PodDisruptionBudget, no anti-affinity / topology-spread**. One Postgres instance per service, **no read replicas, no standby**. Services are stateless by design (JWT/OIDC, Redis for nonce/idempotency) — so this is a wiring gap, not a redesign — but as shipped every service and every DB is a single point of failure.
*Why it matters:* A single pod restart or node drain = payment outage. Cannot scale out under load. This is table-stakes before onboarding volume partners.

### 3. No DR: no multi-AZ/region, no RPO/RTO, no backup/restore drills
No documented RPO/RTO, no backup or restore runbooks, no cross-region replication (MinIO vault has versioning + object-lock but no CRR). AWS overlay pins a single region. ADR-016 gives *application-layer* scheme failover, but there is **no infrastructure DR story**.
*Why it matters:* A money-movement hub with no tested restore path and no recovery objectives is uninsurable and un-auditable by a regulator/partner. "We have backups" ≠ "we have proven we can restore in N hours."

### 4. Schema registry provisioned but bypassed; events are unversioned JSON
Confluent Schema Registry (BACKWARD compat) is in docker-compose + Helm per ADR-001 — but events are published as **plain JSON via `StringSerializer`** (`lib-events-kafka/.../KafkaEventPublisher.java`), and `DomainEvent` is a bare interface with **no version field or schema binding**. The registry is not in the serialization path; nothing enforces compatibility.
*Why it matters:* With many consumers (webhooks, ledger, reporting, partner integrations), an unversioned event contract means a producer change silently breaks downstream consumers at runtime. Governance exists on paper only.

### 5. Distributed tracing is a fire-and-forget tap, not real tracing; no correlation IDs
The custom `trace-console` tap (`lib-errors/.../trace/`) posts raw edges, is **opt-in/off by default, drops events under load, and does NOT propagate trace context.** OTEL endpoint is set in Helm values but **no OTEL SDK/agent is on any service classpath.** **No correlation/request IDs across services**, no RED/USE metrics beyond `/actuator/prometheus` on 2 of 18 services, **no SLOs, no error budgets, no alerting** — only liveness/readiness.
*Why it matters:* When a cross-border payment fails spanning gateway→router→executor→adapter→ledger, there is no way to follow one transaction across services or know if you're breaching an SLO. You are blind at exactly the moment volume exposes you.

### 6. No resilience patterns on inter-service / scheme calls
`resilience4j` appears in **one** file (api-gateway). **No circuit breakers, bulkheads, retry/timeout budgets** on calls to scheme adapters or between services. Gateway rate-limiting exists but is off by default; no per-service back-pressure. No chaos/fault-injection testing anywhere.
*Why it matters:* A slow or down external scheme (ZeroPay/Khalti) will exhaust threads and cascade into a hub-wide outage. External dependencies *will* fail; without breakers/timeouts, one partner's outage becomes everyone's.

### 7. No real integration/E2E environment or contract testing — the H2+fixtures reality
Default tests run on **in-memory H2**; Testcontainers integration is CI-only (no local Docker); dev uses mocked Keycloak (`password=demo`), disabled ZeroPay schedulers, and gated REST clients with fixtures. **No contract testing (Pact/Spring Cloud Contract)** — inter-service coupling rests on shared DTOs only. **No staging environment** — only dev + prod.
*Why it matters:* dev↔prod behavioral drift is guaranteed (H2 vs Postgres, mocked vs real auth/schemes). Bugs surface first in prod. No consumer-driven contracts means service upgrades break each other undetected until integration.

### 8. No analytics/data platform — Finance & partner analytics run on live operational DBs
No warehouse/lake, no CDC (Debezium)/Kafka Connect, no ETL. Reporting and profitability are **ad-hoc SQL against per-service operational databases.** No transaction/settlement **data-retention or archival** policy (vault=10y, audit tiered; transactional = undefined).
*Why it matters:* Analytics queries contend with the payment hot path (a scaling landmine); cross-domain profitability/partner analytics is impossible cleanly across siloed DBs; unbounded operational tables degrade the hot path over time.

### 9. Secrets/key management and PII protection are placeholders
`lib-vault` is a **document object-store (MinIO), not a secrets vault** — misleadingly named. Secrets are K8s Secret placeholders (External Secrets Operator only *recommended* in comments). **No HSM/KMS, no key rotation, no PII tokenization; at-rest PII encryption via HashiCorp Vault is designed (ADR-006) but not implemented** (deferred to R3), as is crypto-shred offboarding.
*Why it matters:* Static shared DB creds, no rotation, and unencrypted PII in transactional stores is a compliance and breach-blast-radius problem for a regulated cross-border processor handling KRW/NPR PII.

### 10. No data-residency / localization enforcement (Korea & Nepal sovereignty)
ADR-016 uses `country` only as a *routing* dimension. All data lands in a **single shared per-service DB regardless of geography** (AWS overlay = Seoul only). No per-region data store, no residence enforcement tying NPR/PII to origin country.
*Why it matters:* Korea (PIPA) and Nepal data-sovereignty rules can require in-country storage. Retrofitting residency after data has commingled across regions is extremely costly — and can block a country launch outright.

### 11. No performance/NFR validation, no capacity plan, no load testing
**No documented NFR targets** (TPS, p99 latency, availability %) beyond a PRD reference; **no Gatling/k6/JMeter, no load tests, no capacity plan.** Hot path is untuned: Hikari at defaults, `@Cacheable` unused, rate quotes cached in-memory (lost on restart), Redis optional/fail-open.
*Why it matters:* Launch (target 2026-10-10) would proceed with **zero evidence** the system meets any throughput/latency bar. You cannot size infrastructure, set SLOs, or promise partners a rate without a measured baseline.

### 12. No progressive delivery, feature flags, or FinOps; scattered tech-debt
No canary/blue-green, no feature-flag system (only API-version toggles), no config service (12-factor env vars only — acceptable but no dynamic control). **No cost tracking/allocation tags/budgets.** Tech debt = 154 TODO/FIXME + implicit WBS phases, **no consolidated debt register**; DLQ replay tooling is designed for webhooks only and not built for transaction/ledger events.
*Why it matters:* Every release is all-or-nothing (no safe rollout/rollback of a bad adapter change); cloud spend is unattributable as you multiply partners/regions; and debt with no register means it's prioritized by memory, not risk.

## CROSS-CUTTING THEME
Roughly half these gaps are **"provisioned but not plumbed"**: schema registry present but bypassed (#4), OTEL endpoint set but no SDK (#5), Vault designed but not integrated (#9), HA-capable Helm but 1 replica (#2), country as a routing tag but no residency (#10). The remediation is often *wiring + hardening*, not rearchitecture — which is good news for cost, but dangerous because status boards mark these "done." **Recommend a go-live readiness gate that distinguishes "code exists" from "path exercised under load in a prod-parity environment."**

## SUGGESTED SEQUENCING (reliability-first)
1. **Pre-launch blockers:** HA (#2 replicas+HPA+PDB), DR runbooks & restore drill (#3), secrets/rotation + PII-at-rest (#9), tracing+correlation IDs+SLOs (#5), resilience4j on scheme calls (#6), NFR targets + one load test (#11).
2. **Scale enablers (next):** Scheme-Adapter SDK + certification harness (#1), schema-registry enforcement (#4), staging env + contract tests (#7), data residency (#10 — before any new country).
3. **Maturity:** analytics/CDC platform (#8), progressive delivery + FinOps + debt register (#12).
