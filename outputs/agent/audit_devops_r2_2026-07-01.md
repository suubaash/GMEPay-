# DevOps / SRE — Second-Pass Audit (post-Operations-backend)

**Scope.** Round 1 (`audit_devops_2026-07-01.md`) audited the platform group (4.21) and found it planned, not built. Since then an Operations backend shipped: `@Scheduled` sweepers (transaction-mgmt `StuckTransactionAlertSweeper`, notification-webhook `WebhookBacklogMonitor`, settlement `ReconBreakAlerter`), Flyway `V038` (config-registry `ops_control`/`ops_suspension`) + `V009` (transaction-mgmt operator-resolution audit), a payment-executor **OperationalGate** that synchronously calls config-registry per authorization (fail-open default), `gmepay.ops.alert` emitters, and ops-partner-bff control-tower + operator actions. This pass sharpens the round-1 platform gaps the ops build made *urgent*, verified against code. Not repeating round-1 verbatim.

---

## (a) Deployment / SRE gaps the ops build newly SHARPENED

### 1. ShedLock is now a HARD replica-count blocker (was: batch-safety follow-up)
Verified: no ShedLock/`@SchedulerLock` anywhere in `code/` (grep hits are only worktrees + docs). The new sweepers are plain `@Scheduled` fixed-delay jobs (`StuckTransactionAlertSweeper.sweep()` line 105; `WebhookBacklogMonitor`; `SettlementGenerationScheduler`/`ReconScheduler`) with **no distributed lock**. Round 1 flagged this generically; the ops build **added three more unguarded schedulers on top of the existing settlement/regulatory/expiry/rotation ones**. On ≥2 replicas every one double-fires: duplicate `ops.alert` storms (operators paged twice), double recon reruns, double regulatory filings/clawbacks. **Consequence: config-registry-gated services cannot run HA. `defaultReplicas: 1` (values.yaml line 26) is now load-bearing — it's the only thing preventing double-fire, which means the whole platform is pinned to single-replica = no HA, no rolling headroom.** This is the #1 go-live blocker the ops build created. Fix: ShedLock on a shared table (or lease) before any `replicas>1`.

### 2. The new alerts have a producer but NO delivery/paging path (detect ≠ wake)
Verified: `gmepay.ops.alert` is emitted via the `EventPublisher` seam → Kafka topic `gmepay.ops.alert`, else **log-only fallback** (`ReconBreakAlerter`, `OpsAlertEvent` route by `eventType=ops.alert`). Grep for a *consumer* of that topic: none. No Alertmanager, no PagerDuty/Opsgenie, no `ServiceMonitor`, no on-call routing (confirmed in chart + round-1 gap #6). **The ops plane can now DETECT incidents (stuck txns, webhook backlog, recon breaks with CRITICAL/WARN severity already classified) but the alert lands in a Kafka topic or a log line that pages no one.** This is the most wasteful gap: real severity-tagged signal is being produced and dropped on the floor. Fix: `ops.alert` → Alertmanager/pager sink + escalation policy; wire the OTLP collector the ABI already references.

### 3. config-registry is now a payment hot-path dependency & new SPOF (fail-open hides its own outage)
Verified: `OperationalGate.checkNewAuthorization()` runs at the START of every new authorization and reads `RestOperationalStatusClient.currentStatus()` → `GET config-registry/v1/ops/operational-status`. Mitigations present and worth crediting: a 3s in-memory TTL cache + last-known-good fallback, so it is **not** a literal per-request round-trip and a brief blip is absorbed. BUT: (a) config-registry is `replicas: 1` with no HA (values.yaml) — its DB/pod outage now touches the pay path; (b) **no explicit connect/read timeout is set on the RestClient** (`RestClientSupport.withJavaTime(...)`) — a *hung* (not refused) config-registry can stall authorizations up to the JDK default until the cache expires; (c) `fail-open` default `true` means a config-registry outage **silently disables the kill-switch** — the one control you most need during an incident is the one that fails off exactly when config-registry is down. Fix: config-registry HA (≥2 replicas — which itself requires gap #1), an explicit short client timeout + circuit-breaker posture, and a decision on fail-open vs fail-closed documented per-env (fail-open is defensible for availability but must be a *conscious* posture, alarmed when engaged).

### 4. New migrations ship in-app-on-boot with no rollback surface (now more blast radius)
Verified: config-registry `spring.flyway.enabled=true` (application.properties line 16) = Flyway runs **inside the app on boot**. `V038`/`V009` ship the same way; there is **no pre-upgrade migration Job, no initContainer, no checksum/gate** in the Helm chart (grep: no `initContainer`). A bad ops migration now downs the service on rollout, and because migration is coupled to app start, `helm rollback` gives you old-code-on-new-schema (round-1 gap #4) — actively unsafe. The ops migrations touch the kill-switch tables (`ops_control`/`ops_suspension`) and the operator-audit table, so a failed migration can take down the incident-control plane itself. Fix: migration-as-pre-upgrade-`Job` (or initContainer) decoupled from app start, with a checksum guard and a paired down-migration/rollback runbook.

### 5. Kill-switch has no operator runbook / console / auth story / gameday
Verified: `ops-partner-bff` `OpsActionController` exposes `POST /v1/admin/ops/pause|resume|maintenance|suspend|unsuspend` → config-registry, audited who/what/when/reason (good). RBAC: requires `ops:operate` **only when `X-Gme-Permissions` header is present; when the header is ABSENT the call is allowed** (dev/gate-off convention). Operationally undefined: *who* invokes pause at 02:00, *from where* (admin-ui exists but no documented pause console/flow), *with what auth path* end-to-end (and the header-absent allow is a mid-incident foot-gun if the edge ever forwards without the header), and *how it's rehearsed* — no gameday, no test that a pause actually stops new authorizations within the 3s cache TTL under load. A kill-switch you've never pulled in anger is not a control. Fix: documented invoke path + auth guarantee (never allow-on-absent-header on the prod edge), an admin-ui pause action wired to it, and a periodic gameday exercising pause→verify-stopped→resume.

### 6. New sweepers & gate are invisible and per-env config is ad-hoc
Verified: chart uses **TCP probes** for transaction-mgmt / notification-webhook / settlement / config-registry / payment-executor (values.yaml) — no `/actuator/health`, and **no Prometheus scrape annotations or ServiceMonitor anywhere** (round-1 gap #5 confirmed still true). So: sweeper last-run/lag, `ops.alert` emit counts, gate call latency/fail-open-engaged counter, and config-registry-unreachable events are **not scraped** — you can't even alert on "the alerter stopped running" or "the gate has been fail-open for 10 min." Config: sweepers are `enabled:false` by default (`gmepay.txn.stuck-alert.enabled`, `GMEPAY_WEBHOOK_DISPATCHER_ENABLED` set true only for notify) — **no per-env values wiring turns the new sweepers on**, so absent explicit config they silently no-op in prod. Secrets: the new cross-service calls (payment-executor→config-registry, ops-bff→config-registry via `GMEPAY_AUTH_IDENTITY_INTERNAL_SECRET` alias) reuse the `CHANGE_ME_*` placeholder secret with no rotation (round-1 gap #11). Fix: metrics + scrape for sweepers/gate, explicit per-env enable flags, real internal-auth secret provisioning.

## (b) Round-1 4.21 items that are genuine GO-LIVE GATES now

1. **Distributed scheduler locking (ShedLock) — gate #1.** Elevated from "batch safety follow-up" to a hard go-live gate: without it you are pinned to single-replica (no HA) or you double-fire regulatory/settlement jobs. The ops build's three new schedulers make this non-optional. Blocks any `replicas>1`.

2. **Alerting / paging / on-call plumbing — gate #2.** The ops build now *produces* severity-classified `ops.alert` signal with no sink. You cannot operate a live money path where incident detection exists but pages no one at 02:00. This is now cheap-to-finish, high-value, and blocking.

3. **Migration-as-pre-upgrade-Job + rollback runbook — gate #3.** In-app-on-boot Flyway now migrates the kill-switch and operator-audit tables; a bad rollout can take down the incident-control plane with no safe rollback. Must be decoupled and rehearsed before shipping ops schema changes to a live path.

(Backup/PITR + DR drill from round-1 gap #9 remains the standing must-fix, but the three above are the ones the ops build specifically converted into blockers.)
